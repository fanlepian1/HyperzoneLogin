# 3. 基础安装

本章目标是帮助你完成一套**最小可运行**的 HyperZoneLogin 部署。由于这是一个模块化框架，所以安装时要先区分：

- 哪个是主插件；
- 哪些是可选模块；
- 哪些依赖需要优先放进 `plugins/`。

## 3.1 安装前确认

在开始之前，请先确认以下前置条件已经满足：

- [ ] 你已经完成了上一章中的 Velocity 与后端服基础配置；
- [ ] 你知道自己要采用哪种登录方案；
- [ ] 你已经准备好 Velocity 代理服运行目录；
- [ ] 你计划把哪些模块放进 `plugins/`。

## 3.2 前置依赖

根据 `用户手册/ai生成词/3基础安装.md` 中给出的安装要求，当前手册保留如下前置说明：

- 先将 `MCKotlinVelocity-版本号.jar` 放入 Velocity 的 `plugins/` 目录；
- 下载地址：<https://github.com/4drian3d/MCKotlin/releases>

仓库内 `gradle/libs.versions.toml` 也有 `# max version of mckotlin-velocity` 的注释，因此可以确认项目与该前置存在关联。实际选择哪个版本，建议以你当前服务端环境与该项目发布说明为准。

## 3.3 插件文件清单

根据工作区 `build/HZL/` 中已有产物，当前可以确认的运行包包括：

- `HyperZoneLogin-1.0-SNAPSHOT.jar` —— 主插件；
- `HZL-auth-offline-1.0.0-SNAPSHOT.jar` —— 离线 / 本地账号模块；
- `HZL-auth-yggd-1.0.0-SNAPSHOT.jar` —— Yggdrasil 在线验证模块；
- `HZL-data-merge-1.0.0-SNAPSHOT.jar` —— 数据迁移模块；
- `HZL-profile-skin-1.0.0-SNAPSHOT.jar` —— 皮肤修复模块。

其中：

- `HyperZoneLogin-1.0-SNAPSHOT.jar` 是基础必需；
- 其他 `HZL-*` 文件按需求选择；
- `api` 项目不是给服主直接放进 `plugins/` 的运行插件。

## 3.4 放置到 `plugins/` 的基本方法

将以下内容放入 Velocity 代理服的 `plugins/` 目录：

1. `MCKotlinVelocity-版本号.jar`
2. `HyperZoneLogin-1.0-SNAPSHOT.jar`
3. 你需要的 `HZL-*` 模块包

## 3.5 不同场景下的最小安装组合

### 方案 A：只做正版 / 第三方 Yggdrasil 验证

至少放入：

- `MCKotlinVelocity-版本号.jar`
- `HyperZoneLogin-1.0-SNAPSHOT.jar`
- `HZL-auth-yggd-1.0.0-SNAPSHOT.jar`

可选：

- `HZL-profile-skin-1.0.0-SNAPSHOT.jar`

### 方案 B：离线 + 正版 / 皮肤站混合登录

至少放入：

- `MCKotlinVelocity-版本号.jar`
- `HyperZoneLogin-1.0-SNAPSHOT.jar`
- `HZL-auth-yggd-1.0.0-SNAPSHOT.jar`
- `HZL-auth-offline-1.0.0-SNAPSHOT.jar`

可选：

- `HZL-profile-skin-1.0.0-SNAPSHOT.jar`

### 方案 C：从旧服迁移

在你目标运行组合的基础上，再加入：

- `HZL-data-merge-1.0.0-SNAPSHOT.jar`

注意：迁移模块不是所有人都要装。它更像是“迁移工具”，而不是“日常运行必备模块”。

## 3.6 首次启动后应出现的内容

根据各模块 `README` 与配置样例，首次启动后你应重点留意以下文件或目录是否生成：

- 运行库缓存：
  - `plugins/hyperzonelogin/libs/`
- 核心配置：
  - `database.conf`
  - `misc.conf`
  - `backend-server.conf`
  - `remap.conf`
- Yggdrasil 配置：
  - `entry/mojang.conf`
  - `entry/` 目录
  - `entry/example/` 示例目录
- 离线识别相关：
  - `offlinematch.conf`
- 迁移模块配置：
  - `merge/merge-ml.conf`
  - `merge/merge-am.conf`
- 皮肤模块配置：
  - `profile-skin.conf`

如果你没有安装对应模块，相关配置目录可能不会出现；这属于正常现象。

另外，当前版本会在首次启动时自动下载若干运行库到 `plugins/hyperzonelogin/libs/`。因此第一次开服时还需要额外确认：

- 代理服能够访问 Maven 仓库镜像；
- 日志中没有出现“依赖校验失败”或“无法加载运行库”之类的报错；
- `libs/` 目录内成功生成若干 `.jar` 缓存文件。

## 3.7 建议的首次启动顺序

为了减少排障难度，推荐这样做：

1. 只放前置与主插件，确认主插件能正常加载；
2. 再加入你真正需要的模块；
3. 首次启动后先不要大改配置，先确认文件成功生成；
4. 检查日志里是否有明显的模块注册失败或主插件缺失提示；
5. 再按下一章逐项理解配置内容。

## 3.8 初次部署时的几个提醒

### 先跑通，再微调

`ai生成词` 希望把文件清单列清楚，目的正是为了让读者先理解“哪些文件负责什么”。初次部署时，最重要的是先确认：

- 插件都被正确识别；
- 配置文件都能成功生成；
- 模块之间能找到主插件并注册；
- 你的登录方案所需模块已经齐全。

### 不要把迁移模块当成功能模块长期保留

如果你只是为了迁移旧数据，可以在迁移完成并验证无误后，再决定是否保留 `data-merge`。对纯日常运行来说，它不是核心依赖。

### 配置文件生成后再修改

很多模块在首次启动时会自动创建带注释的默认配置。与其凭空猜测该写什么，不如先让插件生成原始文件，再按样例逐项调整。

## 3.9 安装完成后的最小验收标准

进入下一章之前，建议至少达到下面这个状态：

- [ ] 代理启动时能识别主插件；
- [ ] 需要的子模块已全部加载；
- [ ] 对应配置文件和目录已经生成；
- [ ] 没有因为缺少前置依赖而直接报错；
- [ ] 你已经清楚自己下一步要改哪些配置，而不是盲目全改。
